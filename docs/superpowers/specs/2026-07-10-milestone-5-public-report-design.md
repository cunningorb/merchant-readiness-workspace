# Milestone 5: Public Report

Date: 2026-07-10

## Context

Milestone 4 (results dashboard) is merged. Per `docs/03_Codex_Execution_Plan.md`, Milestone 5 covers: public report, secure tokens, printable report, optional PDF, with a STOP gate to verify anonymous access.

The `Report` model, migration, and factory already exist (from Milestone 1): one report per assessment (`assessment_id` unique), an auto-generated unique `token`, a nullable `summary`, and a nullable `published_at`. None of it is wired up yet — no controller, no route, no service, and `GET /api/reports/{token}` (a required MVP endpoint per CLAUDE.md) doesn't exist.

Milestone 4 deliberately deferred "how does a merchant come back to view their results later" to this milestone, on the reasoning that a secure report token would solve it without building a separate mechanism. This design resolves that: the report link is surfaced once, right after submit, as the answer.

## Scope

In scope:
- `ReportBuilderService`: creates the `Report` row at submit time and assembles its display payload.
- `SubmitAssessmentService` creates the report as part of the existing submit transaction.
- `GET /api/reports/{report:token}` (JSON) and `GET /reports/{report:token}` (Inertia web page, anonymous-accessible).
- A `report` key (`{ token, url }`) added to the existing submit response, plus a report link surfaced in `AssessmentResults.vue`.
- A new `Reports/Show.vue` page reusing `AssessmentResults.vue` unchanged, with report-specific chrome and a `@media print` treatment for a clean printed/PDF-via-browser output.

Out of scope:
- Any PDF-generation library or dependency — printing is browser-native (`window.print()` + print CSS).
- A separate "publish" step — the report is created and immediately public the moment the assessment is submitted.
- Any narrative/AI-generated `summary` content — the field stays unused (null) this milestone, consistent with the deferred items in GitHub issue #3.
- Any change to `AssessmentResults.vue` itself — it is reused exactly as built in Milestone 4.
- Session/ownership access control on any endpoint — unchanged from the still-deferred Milestone 3/4 discussion. A report's security model is "possession of the token," which is the explicit product requirement ("shareable through secure, non-guessable tokens"), not merchant authentication.

## Backend design

### ReportBuilderService

New service, two methods:

```php
createForAssessment(Assessment $assessment): Report
```
Creates the `Report` row for a just-submitted assessment. `token` is auto-generated by the existing `Report::booted()` hook; `published_at` is set to `now()` immediately — there is no separate publish gate.

```php
buildPayload(Report $report): array
```
Assembles the JSON/props payload for both the API endpoint and the Inertia page:
- The same score data the submit endpoint already returns: `overall_score`, `overall_tier`, `section_scores`, `ranked_sections` (all already stored on the `Assessment` from Milestone 3).
- `recommendations`: the assessment's persisted `Recommendation` rows, same shape as the submit response (`title`, `description`, `category`, `priority`, `expected_impact`).
- A merchant profile context line, built from real, already-stored data (no invented content): `merchant.company_name`, plus three non-scored answers read via `Assessment::answerValue()` — `business.monthly_order_volume`, `catalog.sku_count`, and `platform.ecommerce_platform`. Any of these that are absent (e.g. an older or incomplete assessment) are simply omitted from the line rather than shown as blank/placeholder text.
- `published_at` (for the "Prepared on [date]" header text).

### SubmitAssessmentService integration

Inside `SubmitAssessmentService::submit()`'s existing `DB::transaction()` block (added in the Milestone 3 post-review fix pass), after the recommendation-creation loop and the assessment's `forceFill(...)->save()`, call `ReportBuilderService::createForAssessment($assessment)`. This keeps report creation atomic with the rest of the submit write — if any part of submit fails and rolls back, no orphaned report is created either.

### Endpoints

- `GET /api/reports/{report:token}` — route-model-bound by the `token` column (Laravel's implicit binding via `Route::get('/reports/{report:token}', ...)` with a custom route-key, so an unknown token 404s automatically with no custom lookup code). Returns `ReportBuilderService::buildPayload($report)` as JSON. This satisfies the CLAUDE.md-required endpoint.
- `GET /reports/{report:token}` — a new `ReportController@show`, registered in `routes/web.php` **outside** the `Route::middleware('auth')` group (alongside `/assessment`), so it is reachable anonymously. Renders `Inertia::render('Reports/Show', ['report' => ReportBuilderService::buildPayload($report), 'catalog' => $catalog->sections()])` — the payload is provided server-side as page props, the same pattern the wizard already uses for its `catalog` prop, so the page needs no client-side fetch to the API endpoint above.

### Submit response change

`AssessmentController::submit()`'s JSON response gains one new top-level key:
```json
"report": { "token": "...", "url": "https://.../reports/{token}" }
```
alongside the existing `assessment` and `recommendations` keys. This is the only change to already-shipped Milestone 4 code, and it's the mechanism that resolves the deferred "come back later" question.

## Frontend design

### AssessmentResults.vue

One small, additive change: given a `reportUrl` prop (only passed when rendered from the wizard's post-submit view, not from the report page itself — see below), render a "Your shareable report" link. No other changes — the ring, breakdown, capability mapping, and recommendation cards are reused exactly as built in Milestone 4.

### Reports/Show.vue (new)

A new Inertia page at `resources/js/Pages/Reports/Show.vue`, receiving `report` and `catalog` props from `ReportController@show`. Structure:
- Header chrome: merchant company name as the page title, the profile-context line ("Outdoor Apparel · Shopify · 48,000 annual orders" style, built from whichever of the three optional fields are present), "Prepared on [date]" from `published_at`, and a Print button (`window.print()`).
- `<AssessmentResults :result="..." :catalog="catalog" />` — the same component from Milestone 4, reshaping `report`'s payload into the `result` prop shape it already expects (`{ assessment: { overall_score, overall_tier, section_scores, ranked_sections }, recommendations }`). No `reportUrl` prop passed here — a report page doesn't need a link to itself.
- A `@media print` block scoped to this page (not touching `AssessmentResults.vue`) forcing black-on-white regardless of the app's dark theme, and `print:hidden` on the Print button so it doesn't appear in the printed/PDF output. This avoids depending on the browser's optional "print background graphics" setting and avoids modifying the reused component.

## Testing

Following the project's TDD guardrail for report generation and secure token access:
- `ReportBuilderServiceTest` (unit): `createForAssessment` creates exactly one report per assessment with a non-empty token and `published_at` set; `buildPayload` returns the correct score/recommendation data and correctly includes/omits each optional profile-context field.
- Extend `SubmitAssessmentServiceTest` / `AssessmentSubmissionTest`: submitting a complete assessment also creates a `Report` row and the JSON response includes a `report.token`/`report.url`.
- `ReportAccessTest` (feature): `GET /api/reports/{token}` returns the expected JSON for a valid token and 404s for an unknown one; `GET /reports/{token}` (web) is reachable without authentication (no `auth` middleware applied) and 404s for an unknown token.

Manual verification (no JS test framework in this project, consistent with Milestones 3-4): submit an assessment, follow the report link from the results view, confirm the report page renders the same score/recommendations, confirm it's reachable in an incognito/anonymous browser session (the Milestone 5 STOP gate), and confirm the Print dialog produces a readable black-on-white page.
