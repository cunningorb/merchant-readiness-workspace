# Milestone 7: Demo Data and Reset Command Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the current bare, unanswered demo assessment with three realistic, fully-submitted demo merchants spanning the readiness-tier spectrum, and add a `demo:reset` command that safely regenerates them on demand — in production if needed — without ever touching real prospect data.

**Architecture:** A migration adds an `is_demo` flag to `merchants`, the sole mechanism distinguishing demo rows from real ones. A single seeder class, `Database\Seeders\DemoMerchantsSeeder`, is the one source of truth for demo content: it drives the exact same application services a real merchant's wizard submission uses (`CreateAssessmentService` → `SaveAssessmentAnswersService` → `SubmitAssessmentService`), so scoring/recommendations/reports are generated by real production code, not hand-computed. `DatabaseSeeder` calls it (idempotent, safe on every deploy); a new `demo:reset` Artisan command deletes existing demo rows and calls it again.

**Tech Stack:** Laravel 11 (PHP 8.2), Eloquent, PHPUnit. No new dependency.

## Global Constraints

- No hand-computed scores or hand-inserted `Recommendation` rows anywhere in this milestone — every demo assessment's score/tier/recommendations/report must come from actually calling `SubmitAssessmentService::submit()` (which internally uses `ReadinessScoringService`, `RecommendationEngine`, and `ReportBuilderService`).
- `is_demo` is the only mechanism for identifying demo data. No code in this milestone may identify demo rows by company name, email pattern, or any other convention.
- `DemoMerchantsSeeder` must be idempotent: calling it when any `is_demo` merchant already exists is a no-op. This matches the pattern already used for the admin user in `DatabaseSeeder` (added to fix a production deploy issue — `db:seed --force` now runs on every container start).
- The `demo:reset` command must never require confirmation and must never be capable of deleting a non-demo `Merchant` row, since it may run directly against production.
- Exact target scores (verified against `config/scoring.php` and each `App\Services\Scoring\Questions\*Scorer`): Thistle & Bloom Apparel = 0 (Foundational), Northline Outdoor Supply = 71 (Established), Vantage Home Goods = 100 (Advanced).

`app/Console/Commands/ResetDemoDataCommand.php`:
```php
<?php

namespace App\Console\Commands;

use App\Models\Merchant;
use Database\Seeders\DemoMerchantsSeeder;
use Illuminate\Console\Command;

class ResetDemoDataCommand extends Command
{
    protected $signature = 'demo:reset';

    protected $description = 'Delete and regenerate the three demo merchants (never touches real prospect data)';

    public function handle(): int
    {
        Merchant::where('is_demo', true)->get()->each(fn (Merchant $merchant) => $merchant->delete());

        $this->call('db:seed', [
            '--class' => DemoMerchantsSeeder::class,
            '--force' => true,
        ]);

        $this->info('Demo data reset: 3 merchants regenerated.');

        return self::SUCCESS;
    }
}
```

Laravel auto-discovers command classes under `app/Console/Commands` — no registration step is needed for a Laravel 11 app using the default `bootstrap/app.php` console configuration.
